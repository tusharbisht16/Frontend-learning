Lexical Scope

Lexical scope is the foundational rule of JavaScript — the contract that determines which variables a function can see, decided entirely by where that function is written in the source code.

Definition

Lexical scope (also called static scope) means that the scope of a variable is defined by its physical location in the written source code. The scope is determined at the time the code is parsed — not at the time it is executed. A nested function has access to variables declared in its outer scopes, simply because of where it was written.

Definition
Lexical scope: the scope of an identifier is determined by its position in the source text. The engine resolves names by looking at the textual structure of the program, not the runtime call chain.
Analogy
Think of nested Russian dolls. Each doll can "see" everything inside larger dolls that contain it. But you can never reach into a doll that is beside you — only the ones that physically surround you. The physical nesting structure of the dolls is fixed when they are made, not when you open them.

The Lexical Environment — Where Scope Lives

Every time the engine enters a new scope (global, function, or block), it creates a Lexical Environment — a record that maps identifier names to values. The key to lexical scope is the outer environment reference: a pointer from the inner environment to the environment of the surrounding scope as written in the source.

Lexical environments for nested functions — built from source structure
Global Lexical Environment
greeting = "Hello"  |  outer = [Function]
[[OuterEnv]] → null
outer() Lexical Environment — defined inside Global
name = "Alice"  |  inner = [Function]
[[OuterEnv]] → Global ↑
inner() Lexical Environment — defined inside outer()
suffix = "!"
[[OuterEnv]] → outer() ↑
inner() can access: suffix, name, greeting

Lexical vs Dynamic Scope — The Key Distinction

JavaScript uses lexical scoping. Some other languages (older Lisps, bash, Perl) use dynamic scoping. Understanding the difference is the fastest way to deeply understand lexical scope:

Property Lexical Scope (JavaScript) Dynamic Scope (hypothetical)
Scope determined by… Where the function is written (definition site) Where the function is called (call site)
When is it fixed? At parse/compile time — before execution At runtime — changes with call stack
Predictable? Yes — reading the source is enough No — you must trace the call chain
Enables closures? Yes — outer env ref is permanent No — scope changes per call
lexical-scope-proof.js — call site does NOT affect scope
let value = "global";         // Global lexical env

function readValue() {
  return value;              // Scope chain locked to DEFINITION site
}

function tricky() {
  let value = "tricky local";  // Only exists in tricky's env
  return readValue();         // Calls readValue — does this see "tricky local"?
}

console.log(tricky());        // "global" — NOT "tricky local"
// readValue's scope chain goes: readValue env → Global env
// It was defined in Global scope. tricky's local env is invisible to it.

The [[Environment]] Internal Slot

Every function object has an internal property called [[Environment]]. When a function is created (not called), the engine captures the current lexical environment and stores it in this slot. When the function is later called and a new execution context is made, its outer environment reference is set from [[Environment]]. This is the precise mechanism that implements lexical scope.

The Invariant
A function's outer scope is set once, at the moment the function object is created, and never changes, regardless of how many times the function is called, or from where. This single rule explains hoisting, closures, and the behavior of callbacks.

Scope Chain Walk — Complete Example

scope-chain-walk.js
const planet = "Earth";     // ── level 1: Global

function continent() {
  const region = "Asia";   // ── level 2: continent's env

  function country() {
    const city = "Delhi";  // ── level 3: country's env

    // Lookup chain for each identifier inside country():
    console.log(city);     // found at level 3 ✓
    console.log(region);   // level 3 → level 2 ✓
    console.log(planet);   // level 3 → level 2 → level 1 ✓
  }

  country();
}

Block Scope vs
Function Scope

JavaScript has two fundamentally different scoping mechanisms for variables. Understanding which declaration keyword creates which kind of scope is essential for writing predictable code.

Function Scope — The Original Model

var declarations are function-scoped. A var declared anywhere inside a function is visible everywhere in that function — it belongs to the function's variable environment. Block boundaries (if, for, while, plain {}) have no effect on var scoping.

function-scope-var.js
function demo() {
  if (true) {
    var x = 10;    // "inside" if block — but var ignores blocks!
    var y = 20;
  }
  console.log(x); // 10 — accessible! var belongs to the function
  console.log(y); // 20 — same

  for (var i = 0; i < 3; i++) { /* i is function-scoped */ }
  console.log(i); // 3 — i survives the for-loop block!
}
demo();

Block Scope — ES6 and let/const

let and const are block-scoped. They are confined to the nearest enclosing {} block. The engine creates a separate lexical environment for every block, and let/const live in that block-level environment, invisible outside it.

block-scope-let-const.js
function demo() {
  if (true) {
    let a = 10;    // confined to this if-block
    const b = 20;  // confined to this if-block
    console.log(a, b); // 10 20 ✓
  }
  // console.log(a); // ReferenceError — a doesn't exist here
  // console.log(b); // ReferenceError — same

  for (let i = 0; i < 3; i++) {
    // i exists only inside the for-loop block
  }
  // console.log(i); // ReferenceError — i is gone
}
demo();

Why Block Scope Matters — The For-Loop Problem

The most famous consequence of var's function scope is the classic loop closure bug. This is covered in depth in the Practice section (P4), but here's the setup:

scope-in-loops.js
// ❌ var — single shared binding in function scope
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0);
}
// Logs: 3, 3, 3  ← all share the SAME i

// ✅ let — new binding PER ITERATION (each gets its own scope)
for (let j = 0; j < 3; j++) {
  setTimeout(() => console.log(j), 0);
}
// Logs: 0, 1, 2  ← each closure captures its OWN j
Modern Best Practice
Always default to const. Use let when you need to reassign. Never use var in new code — its function-scope behavior and hoisting to undefined create bugs that are hard to trace. let/const give you predictable block scope, TDZ protection, and no global object pollution.

Closures: Definition
& How They Work

A closure is what happens when a function remembers the variables from the scope where it was born, even after that scope has finished executing. It is not a special syntax — it is an emergent property of lexical scope.

The Formal Definition

Definition
A closure is a function bundled together with references to its surrounding lexical environment (the variable environment of the scope in which it was defined). The function retains access to variables from its enclosing scope even after that scope's execution context has been removed from the call stack.

In practice, this means: when a function is returned from another function, or stored in a variable, callback, or data structure, it carries its scope chain with it. Any variables it referenced in its outer scope remain accessible.

Step-by-Step: How a Closure Forms

▶ Closure Formation — Step 1 of 7

Memory Model — What Actually Persists

When a closure is formed, the outer function's variable environment (lexical environment record) is not garbage-collected, even after the outer function's execution context is popped from the call stack. The inner function object holds a reference to it via its [[Environment]] slot.

The Three Conditions for a Closure

  1. An outer function defines variables in its scope.
  2. An inner function is defined inside the outer function and references those variables.
  3. The inner function is exported (returned, stored, passed as callback) so it outlives the outer function's execution.
closure-anatomy.js — all three conditions annotated
function makeMultiplier(factor) {  // ① outer function — factor is in its env
  return function(number) {       // ② inner function — references outer's "factor"
    return number * factor;
  };                               // ③ inner function is returned → closure formed
}

const triple = makeMultiplier(3); // makeMultiplier EC popped — but factor=3 persists
const quadruple = makeMultiplier(4); // separate closure — factor=4, its own env

console.log(triple(5));    // 15 — finds factor=3 in its closed-over env
console.log(quadruple(5)); // 20 — independent closure with factor=4
console.log(triple(10));   // 30 — triple's factor is STILL 3

Each Closure Gets Its Own Environment

Every call to the outer function creates a new variable environment. Each returned inner function closes over its own independent copy. Closures from different calls to the same outer function are completely separate.

Common Misconception
Closures do not copy the value of variables — they retain a live reference to the variable binding itself. If the outer variable changes after the closure is created, the closure sees the new value. This is why the loop closure problem exists (P4).

Closure Use Cases
in Real Applications

Closures appear throughout professional JavaScript — in event handlers, API factories, memoization, configuration, and virtually every callback ever written. Recognizing these patterns is the mark of an experienced developer.

1 — Function Factories

🏭 Producing specialized functions from a template Very Common
factory.js
function makeValidator(minLength, maxLength) {
  return (input) => {
    if (input.length < minLength) return `Min ${minLength} chars`;
    if (input.length > maxLength) return `Max ${maxLength} chars`;
    return null; // valid
  };
}

const validateUsername = makeValidator(3, 20);
const validatePassword = makeValidator(8, 64);
const validateBio      = makeValidator(0, 160);

console.log(validateUsername("jo"));  // "Min 3 chars"
console.log(validatePassword("secret")); // "Min 8 chars"

2 — Event Handler Configuration

🖱️ Attaching pre-configured data to event listeners Very Common
event-handler.js
function makeClickHandler(buttonId, action) {
  return function() {            // closes over buttonId and action
    console.log(`Button ${buttonId} clicked, action: ${action}`);
    performAction(action);
  };
}

document.getElementById("save-btn")
  .addEventListener("click", makeClickHandler("save", "SAVE_DOCUMENT"));

document.getElementById("delete-btn")
  .addEventListener("click", makeClickHandler("delete", "DELETE_ITEM"));

3 — Memoization (Caching)

⚡ Caching expensive computation results using a closure-held map Performance
memoize.js
function memoize(fn) {
  const cache = new Map(); // ← lives in closure — hidden, persistent

  return function(...args) {
    const key = JSON.stringify(args);
    if (cache.has(key)) {
      console.log("cache hit");
      return cache.get(key);
    }
    const result = fn(...args);
    cache.set(key, result);
    return result;
  };
}

const expensiveFn = memoize((n) => {
  /* slow computation */ return n * n;
});
expensiveFn(10);  // computes → 100, stored in cache
expensiveFn(10);  // "cache hit" → 100

4 — Partial Application & Currying

🔧 Pre-filling some arguments to create specialized functions Functional
partial.js
function partial(fn, ...preset) {
  return function(...later) {  // closes over fn and preset
    return fn(...preset, ...later);
  };
}

const add = (a, b) => a + b;
const add10 = partial(add, 10); // preset = [10] stored in closure
console.log(add10(5));  // 15
console.log(add10(20)); // 30

5 — Callbacks and Async Operations

⏳ Accessing outer scope data inside async callbacks Async
async-closure.js
async function fetchUserPosts(userId) {
  const user = await getUser(userId);

  // This callback closes over "user" and "userId"
  const posts = await getPosts(userId);
  return posts.filter(post => {
    return post.authorId === user.id; // "user" from outer scope
  });
}

Private Variables
Using Closures

JavaScript has no built-in private keyword for regular functions. Closures fill that gap — by keeping variables inside an outer function's scope, they become completely inaccessible from outside, achieving true data privacy.

Why "Private" Matters

In programming, a private variable is one that can only be read or modified through controlled access points — not directly manipulated from outside. Without privacy, any code can overwrite internal state, breaking invariants and causing bugs.

Analogy
A bank account has a balance. If balance were a public variable, any code could write account.balance = 1000000. Instead, you can only interact with it through deposit() and withdraw() — which enforce rules. Closures implement exactly this pattern.

The Pattern: Variables Hidden in Closure Scope

private-counter.js
function createCounter() {
  let count = 0;              // ← PRIVATE: only functions below can touch this

  return {
    increment() { count++; },
    decrement() { count--; },
    reset()     { count = 0; },
    getCount()  { return count; },
  };
}

const counter = createCounter();
counter.increment();
counter.increment();
console.log(counter.getCount()); // 2

// ❌ These all fail — "count" is genuinely inaccessible:
console.log(counter.count); // undefined
counter.count = 999;         // creates new property, doesn't affect the real count
console.log(counter.getCount()); // still 2!

Diagram — Why Outer Code Cannot See Private Variables

Global Scope — what calling code can access
counter = { increment, decrement, reset, getCount }
counter.count → undefined (not a property)
createCounter() Lexical Environment — PRIVATE ZONE
count = 2  (the real value, hidden here)

have [[Environment]] → this scope.
Outside code has NO reference to this environment.

Private Variables with Validation

Private variables let you enforce constraints — ensuring state can only be modified in valid ways:

bank-account.js
function createAccount(initialBalance) {
  let balance = initialBalance;   // PRIVATE
  let transactions = [];          // PRIVATE transaction log

  return {
    deposit(amount) {
      if (amount <= 0) throw new Error("Amount must be positive");
      balance += amount;
      transactions.push({ type: "deposit", amount });
    },
    withdraw(amount) {
      if (amount > balance) throw new Error("Insufficient funds");
      balance -= amount;
      transactions.push({ type: "withdrawal", amount });
    },
    getBalance() { return balance; },
    getHistory() { return [...transactions]; }, // returns a copy!
  };
}

const acct = createAccount(100);
acct.deposit(50);
acct.withdraw(30);
console.log(acct.getBalance()); // 120
// acct.balance → undefined. balance is truly private.

Data Encapsulation
Patterns

Encapsulation is the principle of bundling data with the functions that operate on it, and hiding internal details from the outside world. JavaScript implements it primarily through closures. These are the canonical patterns.

Pattern 1 — The Module Pattern

The original way to create a singleton module with private state and a public API. An IIFE (Immediately Invoked Function Expression) creates a scope, defines private state, and returns a public interface.

module-pattern.js
const ShoppingCart = (function() {     // ← IIFE creates private scope
  // ─── PRIVATE ────────────────────────
  let items = [];
  let discount = 0;

  function calculateTotal() {
    const subtotal = items.reduce((sum, item) =>
      sum + item.price * item.qty, 0);
    return subtotal * (1 - discount);
  }

  // ─── PUBLIC API ─────────────────────
  return {
    addItem(name, price, qty = 1) {
      items.push({ name, price, qty });
    },
    applyDiscount(pct) {
      if (pct < 0 || pct > 1) throw new Error("pct must be 0-1");
      discount = pct;
    },
    checkout() {
      return { items: [...items], total: calculateTotal() };
    },
  };
})();

ShoppingCart.addItem("Book", 20, 2);
ShoppingCart.applyDiscount(0.1);
console.log(ShoppingCart.checkout()); // { items:[...], total: 36 }
// ShoppingCart.items → undefined. Truly private.

Pattern 2 — The Factory Function (Preferred Modern Pattern)

Instead of an IIFE singleton, a factory function creates a new encapsulated instance each time it's called. More flexible than the module pattern and easier to test.

factory-function.js
function createTimer() {
  let startTime = null;      // private
  let elapsed = 0;           // private
  let running = false;       // private

  return {
    start() {
      if (running) return;
      startTime = Date.now();
      running = true;
    },
    stop() {
      if (!running) return;
      elapsed += Date.now() - startTime;
      running = false;
    },
    getElapsed() { return elapsed; },
    reset() { elapsed = 0; running = false; startTime = null; },
  };
}

const t1 = createTimer(); // independent instance, own private state
const t2 = createTimer(); // completely separate — different closure

Pattern 3 — Revealing Module Pattern

A variant where all logic is defined privately, and the return object only exposes references to chosen private functions. Makes the public API very explicit.

revealing-module.js
const AuthService = (function() {
  let currentUser = null;              // private state
  const ADMIN_KEY = "secret123";        // private constant

  function login(user) {                  // private fn
    currentUser = { ...user, loggedIn: true };
  }
  function logout() {                     // private fn
    currentUser = null;
  }
  function isAdmin(key) {                 // private fn
    return key === ADMIN_KEY;
  }
  function getUser() {                    // private fn
    return currentUser ? { ...currentUser } : null;
  }

  // ↓ Only expose what consumers should see
  return { login, logout, isAdmin, getUser };
})();

Pattern 4 — Stateful Singleton (Config Store)

config-store.js
const Config = (function() {
  const store = {};                       // private
  const locked = new Set();              // private — locked keys

  return {
    set(key, value) {
      if (locked.has(key)) throw new Error(`"${key}" is locked`);
      store[key] = value;
    },
    get(key) { return store[key]; },
    lock(key) { locked.add(key); },
  };
})();

Config.set("apiUrl", "https://api.example.com");
Config.lock("apiUrl");
Config.set("apiUrl", "hacked"); // throws Error: "apiUrl" is locked
ES2022 Class Private Fields
Modern JavaScript now has native class private fields using #fieldName syntax. These provide true private state in classes and are now the preferred approach for class-based OOP. However, closures remain the go-to for functional patterns, factories, and any code that doesn't use classes.

🔒 P3-A — Build a stack data structure with private internals
Intermediate

A stack should have push, pop, peek, and size. The internal array should be completely private.

function createStack() {
  const data = [];           // private — cannot be accessed or modified directly

  return {
    push(item) {
      data.push(item);
      return this;
    },
    pop() {
      if (data.length === 0) throw new Error("Stack is empty");
      return data.pop();
    },
    peek() {
      if (data.length === 0) return undefined;
      return data[data.length - 1];
    },
    size() { return data.length; },
    isEmpty() { return data.length === 0; },
  };
}

const s = createStack();
s.push(1).push(2).push(3);
console.log(s.peek()); // 3
console.log(s.pop());  // 3
console.log(s.size()); // 2
console.log(s.data);   // undefined — data is truly private
🔒 P3-B — Rate limiter: block calls that exceed a limit
Advanced

Write a createRateLimiter(maxCalls, windowMs) that returns a function which, when called, returns true if the call is allowed and false if the rate limit is exceeded.

function createRateLimiter(maxCalls, windowMs) {
  const timestamps = [];         // private: log of call times

  return function isAllowed() {
    const now = Date.now();
    const windowStart = now - windowMs;

    // Remove timestamps outside current window
    while (timestamps.length && timestamps[0] < windowStart) {
      timestamps.shift();
    }

    if (timestamps.length < maxCalls) {
      timestamps.push(now);
      return true;   // allowed
    }
    return false;    // rate limited
  };
}

const limiter = createRateLimiter(3, 1000); // 3 calls per second
console.log(limiter()); // true
console.log(limiter()); // true
console.log(limiter()); // true
console.log(limiter()); // false — 4th call in same second

Solving the Loop
Closure Problem

The loop closure problem is one of the most asked JavaScript interview questions. It stems directly from var's function scope. Master all three solutions.

The Problem

loop-bug.js
for (var i = 0; i < 3; i++) {
  setTimeout(function() {
    console.log(i);
  }, i * 100);
}
// Expected: 0, 1, 2
// Actual:   3, 3, 3  ← Why?
Root Cause
var i is function-scoped (to the enclosing function, or global). There is only ONE i binding shared by all three callbacks. By the time the setTimeout callbacks fire (after the loop finishes), i has already been incremented to 3. All three closures reference the same i — and its value is now 3.
Why var produces 3,3,3 — one shared binding
SINGLE var i BINDING
i = 0 → 1 → 2 → 3 (loop ends)
←
callback 0: reads i → 3
callback 1: reads i → 3
callback 2: reads i → 3

Solution 1 — Use let (Modern, Preferred)

✅ Fix it with let — one line change
Beginner
solution-let.js
for (let i = 0; i < 3; i++) {  // ← var → let
  setTimeout(function() {
    console.log(i);
  }, i * 100);
}
// Output: 0, 1, 2  ✓

// Why it works: `let` in a for-loop creates a NEW binding per iteration.
// Each callback closes over its own independent `i`.
// Iteration 0: i=0 (own scope) | Iteration 1: i=1 | Iteration 2: i=2

Solution 2 — IIFE (Classic, Pre-ES6)

✅ Fix it with IIFE — understand why this works
Intermediate
solution-iife.js
for (var i = 0; i < 3; i++) {
  (function(j) {          // IIFE captures current i as parameter j
    setTimeout(function() {
      console.log(j);     // j is local to THIS IIFE call
    }, j * 100);
  })(i);                  // immediately called with current i
}
// Output: 0, 1, 2  ✓

// Why it works: each IIFE invocation creates a NEW execution context.
// i is passed as argument → copied to j in that execution context's VE.
// The callback closes over j (which is a separate binding each time).

Solution 3 — bind / Extra Parameter

✅ Fix it with bind
Intermediate
solution-bind.js
function logNum(n) { console.log(n); }

for (var i = 0; i < 3; i++) {
  setTimeout(logNum.bind(null, i), i * 100);
}
// Output: 0, 1, 2  ✓

// bind(null, i) creates a new function with i pre-filled as first arg.
// The value of i is captured at bind time — a snapshot of the current value.

Closure Memory
Implications

Closures are powerful, but they have a cost: they keep their entire outer variable environment alive in memory. Understanding this prevents memory leaks in long-lived applications.

Why Closures Use More Memory

Normally, when a function returns, its execution context is popped and its local variables are eligible for garbage collection. But when a closure is created, the inner function holds a reference to the outer function's lexical environment — preventing GC from freeing it.

Memory lifecycle — without vs with closure
Without closure — GC frees memory
fn() called → EC created
{ bigData: [...] }
↓
fn() returns → EC popped
No references remain
↓
GC collects bigData ✅
Memory freed
With closure — env persists!
outer() called → EC created
{ bigData: [...] }
↓
Returns inner fn → closure
inner.[[Env]] → outer's VE
↓
outer's VE stays in heap ⚠️
bigData NOT freed

Memory Leak Scenarios

Scenario Why it leaks How to fix
Event listener never removed Closure captures DOM elements or large data; listener stays active indefinitely Call removeEventListener when element is removed from DOM
Closure capturing large object unnecessarily Inner function only needs one field but entire outer object is retained Extract only the needed value into a local variable before the closure
Accumulating closures in array Pushing closures to an array without clearing it — each retains its outer env Clear the array or use weak references (WeakRef, WeakMap) when appropriate
Timers holding closures alive setInterval callback closes over data; timer never cleared Always store and call clearInterval when the timer is no longer needed

Diagnosing Leaks with DevTools

🧪 P5-A — Spot the unnecessary capture
Intermediate
p5a-leak.js
function setup() {
  const hugeArray = new Array(1000000).fill("data"); // 8MB+
  const id = 42;

  // ❌ Problem: closure captures the ENTIRE outer env (including hugeArray)
  return function() {
    console.log(id);  // only needs id — but hugeArray is also retained!
  };
}

const fn = setup();   // hugeArray stays alive as long as fn exists
function setup() {
  let hugeArray = new Array(1000000).fill("data");
  const id = 42;

  // ✅ Fix Option 1: Explicitly nullify large data after use
  processData(hugeArray);      // use it
  hugeArray = null;             // break reference — GC can collect it (needs let, not const)

  return function() {
    console.log(id);
  };
}

// ✅ Fix Option 2: Extract only what the closure needs into a wrapper
function setup() {
  const hugeArray = new Array(1000000).fill("data");
  const id = 42;
  return makeClosure(id);     // pass only id — hugeArray never captured
}

function makeClosure(id) {
  return function() {
    console.log(id);           // closure only sees id, not hugeArray
  };
}

// Note: modern V8 is smart and may not retain hugeArray if it's provably
// unreachable from the closure, but don't rely on engine optimizations.
🧪 P5-B — Fix the timer leak
Advanced
p5b-timer-leak.js
function startPolling(url) {
  const state = { active: true, data: [] };

  // ❌ Interval never cleared — closure and state leak forever
  setInterval(async () => {
    const result = await fetch(url);
    state.data.push(await result.json());
  }, 5000);
}
function startPolling(url) {
  const state = { active: true, data: [] };

  // ✅ Store the interval ID so it can be cleared
  const intervalId = setInterval(async () => {
    if (!state.active) {
      clearInterval(intervalId);   // clean up if deactivated
      return;
    }
    const result = await fetch(url);
    state.data.push(await result.json());
  }, 5000);

  // ✅ Return a stop function — callers can stop polling
  return function stop() {
    state.active = false;
    clearInterval(intervalId);
    state.data = [];   // optionally clear data too
  };
}

const stopPolling = startPolling("/api/data");
// Later, when no longer needed:
stopPolling(); // cleans up interval and state

Using DevTools Memory Profiler

In Chrome DevTools → Memory tab, you can take heap snapshots before and after operations. Look for Closure entries in the snapshot — clicking them shows what variables they're retaining and the size. If you see an unexpectedly large closure, that's your leak.

General Rules to Avoid Closure Memory Issues
  1. Don't capture variables you don't need — extract only required values before creating the closure
  2. Always clear setInterval and setTimeout when done
  3. Remove event listeners when their elements are removed from the DOM
  4. Set large variables to null after use if a closure will outlive them
  5. Prefer factory functions over long-lived singletons with closures holding large state
