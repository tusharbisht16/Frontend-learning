JavaScript Execution
Model Overview

Before a single line of your code runs, JavaScript has already done considerable work. Understanding this pipeline — from source text to running instructions — is the foundation of everything else.

What "Execution Model" Means

The execution model describes the complete system JavaScript uses to run your code: how it is received, how environments are set up, how functions are tracked, and how control flows between them. It is the mental model that explains why code behaves the way it does.

Definition
The JavaScript execution model is the set of rules governing how the engine receives source code, creates execution environments, manages the call stack, and ultimately produces observable output.

The Three Pillars

The entire execution model rests on three core mechanisms. Every behavior you will ever observe in JavaScript — hoisting, closures, this binding, scope — can be traced back to these:

Three pillars of the JavaScript execution model
Pillar 1
Execution Contexts
Containers that hold code's runtime environment: variables, functions, scope, and this. Created for every function call.
Pillar 2
The Call Stack
A LIFO data structure tracking which context is currently executing. Determines control flow and manages context lifetimes.
Pillar 3
The Scope Chain
A linked chain of variable environments allowing inner scopes to access outer variables. Built at function creation, not at call time.

Single-Threaded and Synchronous — By Default

JavaScript engines execute code on a single thread. This is fundamental: only one piece of code runs at any given moment. There is one call stack. There is no parallel execution of synchronous code.

Critical Implication
Because JS is single-threaded, if a function takes a long time, everything else waits. The call stack is blocked. This is why long-running operations use callbacks, promises, and async/await — to defer work to the event loop rather than blocking the main thread.

Before your code produces any output, it goes through a pipeline inside the JavaScript engine (e.g. V8, SpiderMonkey, JavaScriptCore):

JavaScript is Interpreted — But Also Compiled

JavaScript is often called an "interpreted language", but modern engines like V8 use Just-In-Time (JIT) compilation. The source code is parsed into an AST, compiled to bytecode, profiled for hot paths, and then optimized machine code is generated for frequently-run sections.

For understanding the execution model, the key practical takeaway is that the engine sees the entire source before executing. This is why hoisting and the creation phase exist.

Analogy
Think of the JavaScript engine as a chef who reads the entire recipe before starting to cook. During the read-through (creation phase), they gather all ingredients and prep the station. Only then do they start cooking (execution phase), step by step.

The Big Picture: What Happens When You Run a Script

example.js — traced step by step
var userName = "Alice";          // ← assigned in execution phase

function greet(name) {          // ← stored whole in creation phase
  return "Hello, " + name;
}

var message = greet(userName); // ← new function EC pushed onto stack
console.log(message);           // ← outputs "Hello, Alice"

Execution sequence for this script:

  1. Engine receives source text and tokenizes + parses it into an AST
  2. Global Execution Context created (Creation Phase): userName → undefined, greet → [Function], message → undefined
  3. Global EC execution begins: userName assigned "Alice"
  4. greet(userName) called → new Function EC pushed to call stack
  5. greet's Creation Phase: parameter name → "Alice"
  6. greet's Execution Phase: returns "Hello, Alice" → EC popped
  7. message assigned "Hello, Alice"
  8. console.log called → outputs Hello, Alice

Global Execution
Context

The Global Execution Context is the foundation upon which all JavaScript runs. It is created automatically when a script loads, and it exists until the script terminates.

What Is the Global EC?

When the JavaScript engine begins running a script, the very first thing it does is create the Global Execution Context (GEC). This happens before any code is executed. There is exactly one GEC per script, and it lives until the browser tab is closed or the Node.js process exits.

Definition
The Global Execution Context is the default execution environment in which JavaScript code runs when it is not inside any function. It is the base context, always present at the bottom of the call stack.

What the GEC Contains

Every execution context is composed of three components. For the Global EC, they are set up as follows:

Global Execution Context — internal structure
🌍 GLOBAL EXECUTION CONTEXT
Variable Environment
All var declarations, function declarations at global scope
Scope Chain
Points to its own Variable Environment + null (no outer scope)
this
Browser: window object  |  Node.js: global object (or {} in modules)
Global Object
Created alongside GEC. All global vars become properties of it.
Outer Environment
null — there is nothing outside global scope

The Global Object

Alongside the GEC, the engine creates the global object (window in browsers, global in Node.js). All variables declared with var at the global level become properties of this object. Function declarations also become properties of the global object.

global-object.js — browser environment
var age = 25;
function sayHi() { return "hi"; }

console.log(window.age);   // 25 — var becomes window property
console.log(window.sayHi); // [Function: sayHi] — same here

let name = "Bob";
console.log(window.name);  // "" (window.name is a built-in, let/const don't attach to global obj)
Important Distinction
var declarations in global scope become properties of the global object. let and const declarations in global scope do not — they exist in a separate "script scope" or declarative environment record that is not the global object.

Two Phases of the GEC

Like all execution contexts, the GEC is created in two phases:

Phase 1
Creation Phase
① Create global object
② Set this = global obj
③ Scan for var → set to undefined
④ Scan for function decls → store complete function
⑤ Build scope chain
⑥ Allocate heap memory for objects/arrays
Phase 2
Execution Phase
① Execute code top-to-bottom
② Assign actual values to var declarations
③ Execute expressions
④ Push new EC for each function call
⑤ Pop EC when function returns
⑥ Continue until script end

Function Execution
Context

Every time a function is invoked, JavaScript creates a brand-new execution context just for that call. Understanding this context — what it contains, when it's created, and when it's destroyed — is the key to understanding scope, closures, and this.

Created On Every Call, Not On Every Definition

A critical distinction: the Function Execution Context (FEC) is created when a function is called, not when it is declared. Every invocation of a function creates a fresh, independent execution context, even if the same function is called multiple times.

multiple-calls.js
function add(a, b) {
  let result = a + b;
  return result;
}

add(1, 2);  // ← EC #1 created: {a:1, b:2, result:undefined→3}
add(5, 6);  // ← EC #2 created: {a:5, b:6, result:undefined→11}
add(1, 2);  // ← EC #3 created: completely independent of EC #1

// Each EC is separate. EC #1 and EC #3 have no connection.

Structure of a Function EC

Every Function EC contains the same fundamental components as the Global EC, but with a key addition: it has an outer environment reference pointing to the scope where the function was defined.

Function Execution Context — full internal structure
⚙️ FUNCTION EXECUTION CONTEXT — greet("Alice")
Arguments Object
{ 0: "Alice", length: 1 } — array-like, holds passed args
Parameters
name = "Alice" — treated like local variables
Variable Environment
All var/let/const declared inside the function body
Scope Chain
Own VE + outer ref → ... → Global VE
Outer Environment Ref
Points to the Variable Environment of the scope where this function was lexically defined
this
Determined by how the function is called (see table below)

Lifetime of a Function EC

Function EC lifecycle: from call to garbage collection
1. FUNCTION CALLED → Engine creates new Function EC object in memory
2. CREATION PHASE → Vars hoisted, scope chain set, this bound, arguments built
3. PUSHED TO STACK → EC placed on top of call stack. Becomes the "currently running" context
4. EXECUTION PHASE → Code inside function runs line by line. Values assigned. Calls nested functions.
5. RETURN / END → Function returns (explicitly or implicitly returns undefined)
6. POPPED FROM STACK → EC removed from call stack. Previous EC resumes.
7. GC ELIGIBLE* → EC memory freed unless a closure holds a reference to its variable env
Note on Closures
Step 7 has an asterisk for a reason. If a function defined inside another function still holds a reference to the outer function's variable environment (via the scope chain), that variable environment cannot be garbage collected. This is the mechanism behind closures.

this in Function Contexts

Unlike the Global EC where this is always the global object, the value of this inside a function depends entirely on how the function is called:

Call Syntax this value Example
Regular function call undefined (strict) / window (non-strict) fn()
Method call The object before the dot obj.method()
new call The newly created object new Fn()
call / apply First argument passed explicitly fn.call(ctx)
Arrow function Inherited from enclosing lexical context (no own this) const f = () => {}

Call Stack Mechanics
& Visualization

The call stack is the engine's bookkeeper — it tracks exactly which function is running and what to do when it finishes. Mastering the call stack means you can mentally simulate any JavaScript program's execution.

What Is the Call Stack?

The call stack (also called the execution stack or runtime stack) is a data structure that records the order in which functions are called. It works as a strict Last In, First Out (LIFO) stack: the most recently called function is always the one currently executing, and when it returns, the previous function resumes.

Analogy
Imagine a stack of plates. You can only add to the top or remove from the top. When greet() calls getName(), getName() goes on top. When getName() returns, it's lifted off and greet() is on top again. The Global EC is the heavy plate always at the bottom.

The Four Rules of the Call Stack

Rule 1
Global EC starts at bottom
The Global EC is always the first frame. It is never popped until the script ends.
Rule 2
Every call pushes a frame
When a function is invoked, a new EC is created and pushed onto the stack.
Rule 3
Only the top frame executes
The function at the top of the stack is the "currently executing" context. All others are paused.
Rule 4
Return pops the frame
When a function returns, its EC is removed. The frame below it becomes the new top and resumes.

Stack Overflow

The call stack has a finite size (typically ~10,000–15,000 frames depending on the browser and system). If function calls keep pushing frames without any returning (usually due to infinite recursion), the stack fills up and the engine throws a RangeError: Maximum call stack size exceeded.

stack-overflow.js
function recurse() {
  recurse(); // no base case — pushes forever
}
recurse(); // → RangeError: Maximum call stack size exceeded

// Fix: always have a base case in recursive functions
function countDown(n) {
  if (n === 0) return;      // ← base case stops recursion
  console.log(n);
  countDown(n - 1);
}

Reading Call Stack in DevTools

In Chrome/Firefox DevTools, when you hit a breakpoint or an error, the Call Stack panel shows the live call stack. The topmost entry is the currently paused function. Reading from top to bottom traces the chain of calls that led here.

DevTools Tip
Place a debugger; statement in your code. When DevTools is open, execution pauses there and you can inspect the full call stack, local variables for each frame, and step line by line. This is the most powerful way to build intuition for the execution model.

How JavaScript Code
Is Parsed & Executed

Your source code text travels through a multi-stage pipeline before a single instruction runs. Understanding this pipeline explains syntax errors, hoisting, and the difference between compile-time and runtime behavior.

Stage 1 — Tokenization (Lexing)

The raw source code string is fed into the lexer (tokenizer). The lexer reads characters left-to-right and groups them into meaningful tokens — the atomic units of the language.

Source text → token stream
Source: var x = 5 + 3;

Stage 2 — Parsing (AST Construction)

The token stream is fed into the parser, which checks tokens against JavaScript's grammar rules and constructs an Abstract Syntax Tree (AST). The AST is a tree-shaped representation of the code's structure, with no extraneous syntax characters.

Syntax errors are caught here. If the grammar rules are violated (e.g. var = 5), the parser throws a SyntaxError before any code runs.

Why This Matters for Hoisting
The parser sees the entire source before execution begins. During AST construction, the engine knows every variable and function declaration in each scope. This scan result is what enables the creation phase to hoist declarations before code runs.

Stage 3 — Compilation to Bytecode

Modern engines (V8, SpiderMonkey) don't interpret the AST directly. They compile it to an intermediate bytecode representation, then execute that. Hot code paths are further compiled to optimized machine code by a JIT compiler (V8 uses Ignition + TurboFan).

As a developer, you don't interact with this stage directly, but it's why JavaScript performance has improved dramatically — the engine can optimize repetitive patterns.

Stage 4 — Execution

Now the compiled code actually runs. The engine:

  1. Creates the Global EC (with Creation Phase)
  2. Pushes it to the call stack
  3. Begins executing the bytecode top-to-bottom
  4. For each function call: creates a new EC, pushes it, runs it, pops it on return

Syntax Errors vs Runtime Errors

Because parsing happens before execution, understanding which errors come from which stage is important:

Error Type When Caught Example Entire Script Runs?
SyntaxError Parsing stage — before any line runs var = 5; / unclosed { No — nothing executes
ReferenceError Execution stage — at the problematic line console.log(undeclaredVar) Yes, up to that line
TypeError Execution stage — at the problematic line null.property Yes, up to that line
RangeError Execution stage — when limit exceeded Stack overflow / invalid array length Yes, up to that point

The Event Loop Connection

The execution model described so far covers synchronous code. But JavaScript also handles asynchronous operations. The event loop sits outside the call stack and feeds callbacks into it when the stack is empty. Understanding the call stack makes the event loop intuitive: async callbacks never interrupt running code, they wait for the stack to clear.

Scope of This Guide
The event loop is a natural continuation of the execution model study. It's not covered here but is covered in the JS concurrency model — worth studying after mastering these 8 topics.

Creation Phase vs
Execution Phase

Every execution context — global or function — is created in two passes over the code. The first pass sets up the environment. The second pass runs the instructions. Hoisting, temporal dead zones, and closure binding all flow from these two phases.

The Two-Phase Model

When an execution context is created (whether for the global scope or a function call), the engine performs two distinct passes:

Two-phase execution context creation
Phase 1 — Creation
Memory Setup

The engine scans the entire function body (or script) and sets up memory before running any code:

  • var declarations → created, initialized to undefined
  • let / const → created, placed in Temporal Dead Zone (uninitialized)
  • Function declarations → stored completely (name + body)
  • Function parameters → created, assigned argument values
  • Outer environment reference → set (scope chain begins)
  • this → bound
  • arguments object → built (function ECs only)
Phase 2 — Execution
Code Runs

The engine runs through the code top to bottom, line by line:

  • var assignments → actual values written over undefined
  • let / const → initialized when declaration line reached
  • Expressions evaluated
  • Conditions tested, loops executed
  • Function calls → new EC created, pushed to stack
  • Return values propagated back to caller

Hoisting — The Visible Effect of Creation Phase

Hoisting is the name given to the observable behavior that arises from the creation phase. It's not that declarations are physically moved — it's that they are registered in memory before the execution phase begins, making them accessible (in some form) before their written position in the source.

hoisting-demo.js — creation phase shown first
// ── CREATION PHASE of Global EC ──────────────
// Engine scans and sets up in memory:
// greeting    → undefined  (var hoisted)
// result      → undefined  (var hoisted)
// message     → [TDZ]      (let — not initialized)
// sayHello    → [Function] (function declaration — fully stored!)

// ── EXECUTION PHASE begins ───────────────────
console.log(sayHello());   // "Hello!" — works! fn was in memory
console.log(greeting);    // undefined — var exists but not assigned yet
// console.log(message);  // ReferenceError — let is in TDZ

var greeting = "Hi";        // now greeting = "Hi"
let message = "Hey";         // now message initialized = "Hey"
var result = sayHello();    // "Hello!" — still works

function sayHello() {
  return "Hello!";
}

Temporal Dead Zone (TDZ) — A Creation Phase Artifact

The Temporal Dead Zone is the period between the start of a block and the point where a let or const declaration is reached. During this period, the binding exists in the variable environment (created in the creation phase) but is not initialized. Accessing it throws a ReferenceError.

The Critical Insight

Core Mental Model
The creation phase explains why you can call a function declaration before its written position (it was fully stored in phase 1), why var variables appear as undefined before their assignment (hoisted but not yet assigned), and why let/const throw errors before their declaration (TDZ, not yet initialized in phase 1).

Variable Environment
Concept

The variable environment is the actual data structure where identifiers and their values live inside an execution context. Understanding its internal mechanics explains identifier resolution, closures, and block scoping.

What Is the Variable Environment?

Every execution context contains a variable environment — an object-like structure that stores all the variable bindings (identifier → value mappings) for that context. When you declare a variable or function, you are adding an entry to the variable environment of the current execution context.

Specification Term
In the ECMAScript specification, this is formally called an Environment Record. The Variable Environment is a reference to an Environment Record. Modern JS (ES6+) actually has two records per context: the Variable Environment (for var) and the Lexical Environment (for let/const and blocks). Conceptually, treat them as "the place where this context's variables live."

Environment Record Types

The spec defines several types of environment records, each serving a different purpose:

Record Type Where Used What It Stores
Global Environment Record Global EC Global var bindings (Object Record) + let/const (Declarative Record) + global object
Function Environment Record Function EC Parameters, local var/let/const, function declarations, arguments, this
Block Environment Record Block {} with let/const Only let/const declared inside that block
Module Environment Record ES Module Module-level bindings, import bindings (live links to other modules)

var vs let/const — Different Records

ES6 introduced an important distinction. Inside a function context, there are conceptually two separate environment records:

  • The Variable Environment — holds var declarations (function-scoped)
  • The Lexical Environment — holds let/const declarations and block scoping

This is why var ignores block boundaries (if, for, while) while let/const respect them — they live in different record structures.

var-vs-let-scope.js
function demo() {
  if (true) {
    var a = 1;   // goes into function VE → visible whole function
    let b = 2;   // goes into block Lexical Env → dies after block
    const c = 3; // same as let
  }
  console.log(a); // 1 — var escapes block into function VE
  console.log(b); // ReferenceError — b only existed in block VE
}

Closures — The Variable Environment Persists

A closure occurs when a function retains a reference to the variable environment of its enclosing scope, even after that scope's execution context has been popped from the call stack. The VE is not garbage collected because the inner function's scope chain holds a live reference to it.

closure-venv.js
function makeAdder(x) {
  // makeAdder's VE: { x: 5 } ←────────────────────┐
  return function(y) {                           │
    return x + y;  // x found via scope chain ──────┘
  };
}

const add5 = makeAdder(5);
// makeAdder EC is popped from stack here
// BUT: add5's [[OuterEnv]] still points to makeAdder's VE
// makeAdder's VE {x:5} cannot be garbage collected

console.log(add5(3));  // 8 — x=5 still accessible
console.log(add5(10)); // 15
Memory Implication
Every closure keeps its outer variable environment alive in memory. Creating many closures that capture large environments (objects, arrays, etc.) can be a source of memory pressure in long-lived applications. This is why you should avoid capturing unnecessarily large objects in closures.

Scope Chain
Formation

The scope chain is the mechanism that allows JavaScript to resolve identifiers across nested functions. It is set at function definition time — not at call time. This is the engine's implementation of lexical scoping.

What Is the Scope Chain?

The scope chain is a linked list of variable environments that the engine walks when resolving an identifier. Every execution context has a scope chain built from:

  1. Its own variable environment (searched first)
  2. Its outer environment reference → that environment's VE
  3. That environment's outer reference → …
  4. Until the Global VE is reached
  5. If not found → ReferenceError
Lexical Scoping
JavaScript uses lexical scoping (also called static scoping): the scope chain of a function is determined by where the function is written in the source code, not where it is called from. A function's scope chain is fixed at the point of its creation.

Scope Chain Formation — Step by Step

When the engine creates a function (during the creation phase of the enclosing EC), it attaches a [[Environment]] internal slot to the function object. This slot holds a reference to the variable environment of the scope where the function was created. When the function is later called, its new EC's outer environment reference is set to the value of [[Environment]].

Scope chain formation for nested functions
Global Scope
x = 1  |  outer = [Function]
[[OuterEnv]] → null
outer() Scope — created inside Global
y = 20  |  inner = [Function]
[[OuterEnv]] → Global Scope ↑
inner() Scope — created inside outer()
z = 30
[[OuterEnv]] → outer()'s Scope ↑
Looking up x from here:
① inner's own env → not found
② outer()'s env → not found
③ Global env → found! x = 1 ✅

Lexical vs Dynamic Scoping — Why It Matters

JavaScript's lexical scoping means the scope chain is determined by the code's structure, not the call stack at runtime. This is different from dynamic scoping (which some other languages use) where the scope chain follows the call chain.

lexical-scope-proof.js
let value = "global";

function getScope() {
  return value;    // scope chain set at DEFINITION — sees "global"
}

function caller() {
  let value = "local";  // this does NOT affect getScope
  return getScope();     // calls getScope from inside caller
}

console.log(caller()); // "global" — NOT "local"!
// Proof: getScope's scope chain was locked at definition time
// It looks up its own chain (→ Global), not caller's chain
The Golden Rule of Lexical Scope
A function's scope chain is determined by where it was written, not where it is called from. Even if you call a function from a deeply nested context, it will only "see" the variables in the scopes that surrounded it at the point of its definition.

Identifier Resolution Algorithm

When the engine encounters an identifier (like x), it runs this algorithm:

Identifier resolution — step-by-step algorithm
Encounter identifier x
Search current EC's Variable Environment
FOUND → return value ✅
NOT FOUND → follow [[OuterEnv]]
Search outer VE
(repeat until found or null)
Outer is null (Global reached) → ReferenceError

Variable Shadowing

When an inner scope declares a variable with the same name as an outer scope, the inner variable shadows the outer one. The outer variable is not modified — it simply isn't reached during identifier resolution because the search stops at the inner scope.

shadowing.js
let color = "red";    // Global VE: { color: "red" }

function paint() {
  let color = "blue";  // paint's VE: { color: "blue" } — SHADOWS global
  console.log(color);   // "blue" — resolution stops at paint's VE
}

paint();
console.log(color);    // "red" — global color unchanged, untouched

Block Scope and the Scope Chain (ES6+)

With let and const, every block ({}) creates its own lexical environment that is added to the scope chain. This means the scope chain can include block-level environments between function environments.

block-scope-chain.js
function example() {       // Function VE: { a: 1 }
  let a = 1;
  {                         // Block VE: { b: 2 }  outer → Function VE
    let b = 2;
    {                       // Block VE: { c: 3 }  outer → Block VE above
      let c = 3;
      console.log(a, b, c); // 1 2 3 — chain: innerBlock→outerBlock→function
    }
    // console.log(c); // ReferenceError — c's block closed
  }

When is it formed? At function definition time — the [[Environment]] slot is set when the function object is created
What does it contain? A linked list of Variable Environments from current → outer → ... → Global
How is it walked? Identifier lookup searches inward-to-outward, stopping at first match
Can it be modified at runtime? No. The chain structure is fixed. Values within environments can change, but the chain links cannot.
What enables closures? The outer environment reference keeps outer VEs alive even after their ECs are popped
Does call location matter? Never — lexical scope means only definition location matters
